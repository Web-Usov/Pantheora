## Главный принцип

Сначала сохраняй рабочий baseline, потом экспериментируй.

Это означает:

- не ломай рабочий `Paperclip + OpenClaw` стенд без явной причины
- новые гипотезы лучше выносить в отдельные compose-файлы, профили или docs
- если меняешь интеграционный контракт, обязательно обнови документацию

## Что сейчас считается baseline

Baseline на текущий момент:

- основной compose: `docker-compose.yml`
- вспомогательный plain OpenClaw compose: `docker-compose.openclaw.yml`
- bootstrap-патч Paperclip: `config/paperclip/boot-patched-paperclip.mjs`
- базовый OpenClaw config: `config/openclaw/openclaw.json`

Не удаляй и не переписывай эти артефакты без понимания, зачем это делается.

## Работа с Docker

Основные команды:

```bash
docker compose up -d
docker compose ps
docker compose logs -f paperclip openclaw-gateway
docker compose down
```

Правила:

- предпочитай `docker compose` вместо ad-hoc команд
- перед выводами о поломке проверяй `docker compose ps` и healthchecks
- если проблема касается межконтейнерной связности, помни:
  - `localhost` внутри контейнера почти всегда значит сам контейнер
  - для Paperclip из других контейнеров нужен `http://paperclip:3100`

## Работа с данными и секретами

Никогда не коммить:

- `.env`
- содержимое `data/`
- реальные токены, API keys, claimed keys, device keys

В репозитории можно хранить только:

- `.env.example`
- шаблонные конфиги
- код bootstrap/patch логики
- документацию без секретов

Если в логах или JSON встречается токен, не вставляй его в docs и commit message.

## Работа с кодом и конфигами

При правках:

- предпочитай маленькие целевые изменения
- не делай "большую уборку" заодно
- не меняй версии образов без причины и без фиксации в docs

Если приходится патчить сторонний код на старте контейнера:

- объясни в docs, зачем нужен патч
- укажи, что именно несовместимо в upstream
- по возможности делай патч идемпотентным

## Документация обязательна

После любого значимого исследования или интеграционного вывода:

- обнови `docs/current-state.md`, если поменялся реальный статус системы
- обнови `docs/paperclip-openclaw.md`, если поменялись шаги запуска или ограничения
- при появлении новой execution-попытки добавь отдельный doc в `docs/runs/`

Не оставляй критичные знания только в чате или только в git diff.

### Как раскладывать документацию

- `README.md` — общее видение проекта
- `docs/current-state.md` — общий статус и карта экспериментов
- `docs/paperclip-openclaw.md` — quickstart и техничка для конкретного стенда
- `docs/runs/*.md` — наблюдения, выводы и итоги по отдельным попыткам

Не смешивай общую картину проекта с выводами только по одной вариации.

## Как оценивать результат

Недостаточно, чтобы сервис "открылся в браузере".

Для интеграционных задач старайся подтверждать:

- контейнеры `healthy`
- API доступны
- реальный рабочий сценарий проходит end-to-end
- actor/identity корректны, если речь про агентское выполнение

Для `Paperclip` особенно важно отличать:

- просто успешный heartbeat
- корректное выполнение задачи нужным агентом
- полноценную управляемость агента из UI

## Ограничения текущей архитектуры

На данный момент считай рабочей гипотезой:

- `openclaw_gateway` полезен как внешний runtime bridge
- но это не идеальный основной локальный агент Paperclip

Если исследуешь `OpenCode` или другой runtime, сравнивай не только "запускается / не запускается", но и:

- есть ли `instruction bundles`
- есть ли `skill sync`
- насколько агент управляем из Paperclip
- создаёт ли он реальные сущности в системе, а не только текстовые отчёты

## Git-практика

Предпочтительно:

- отдельный коммит на отдельный смысловой шаг
- понятные commit messages
- не смешивать документацию, refactor и новый эксперимент без необходимости

Если сделал новый инфраструктурный baseline, закоммить его до следующего риска.

## Работа с субагентами

Для этого репозитория допустим и желателен разделённый режим работы:

- основная сессия держит архитектуру, интеграционный контекст и общую траекторию эксперимента
- субагенты получают узкие, хорошо ограниченные технические подзадачи

Рекомендуемый паттерн:

- главная сессия работает как "архитектор"
- техническое исполнение можно делегировать субагентам
- если полезно, субагенту можно назначать другую модель под тип задачи

Базовая рекомендация по моделям:

- `gpt-5.4` — для архитектуры, синтеза, спорных интеграционных решений и общей координации
- `gpt-5.3-codex` — для конкретных инженерных задач, code-level исследования, правок compose/config/bootstrap, локальных реализаций

Когда стоит делегировать:

- задача чётко ограничена по ответственности
- подзадачу можно выполнить параллельно с основной работой
- нужен отдельный технический проход по конкретной зоне

Когда не стоит делегировать:

- подзадача неотделима от основного решения
- следующий шаг целиком зависит от её результата прямо сейчас
- есть риск дублирования или конфликтующих правок

Важно:

- субагентам надо явно задавать область ответственности
- не отправлять несколько субагентов править один и тот же участок без необходимости
- результат субагента считать частью общего процесса, а не отдельной истиной

## Если не уверен

Если видишь конфликт между "быстро попробовать" и "не потерять рабочее состояние", выбирай путь с меньшим риском:

- сначала документируй
- потом выделяй эксперимент отдельно
- потом проверяй end-to-end

Цель этого репозитория сейчас не в идеальной чистоте, а в накоплении воспроизводимого знания и работающих сценариев.
